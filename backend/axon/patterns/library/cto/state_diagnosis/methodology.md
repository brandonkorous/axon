# State Diagnosis

Will Larson identifies four states an engineering team can be in: falling behind (backlog grows, morale drops), treading water (work gets done but no improvement), repaying debt (investing in foundations at the cost of features), and innovating (shipping features on a healthy codebase). Each state requires a different intervention — applying the wrong fix to the wrong state makes things worse.

Apply this pattern when deciding how to allocate engineering effort. Before adding features, hiring, or restructuring, first diagnose the current state. A team that is falling behind needs headcount or scope reduction, not a refactoring initiative. A team that is treading water needs permission to repay debt, not more feature pressure.

**Steps:** (1) Assess the team honestly — is the backlog growing, stable, or shrinking? (2) Ask whether the team can absorb a new feature request without slipping. (3) Classify the state: falling behind, treading water, repaying debt, or innovating. (4) Match your intervention to the state. (5) Reassess quarterly — the goal is to move every team toward the innovating state over time.

The most common mistake is demanding innovation-state output from a team that is falling behind. The second mistake is never allowing a team in the repaying state to stop repaying and start building again.
